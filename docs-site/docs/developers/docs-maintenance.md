# Docs maintenance

The documentation site lives in `docs-site/`. Some pieces are
hand-written prose; others are generated from the runtime so they
cannot drift. This page explains how the moving parts fit together
and what to update when you change code that the docs reference.

## Site layout

```text
docs-site/
├── docs/
│   ├── users/        — user guide pages
│   ├── developers/   — internals + maintenance (this file)
│   ├── reference/    — auto-derivable reference pages
│   └── adr/          — Architecture Decision Records
├── src/
│   ├── pages/        — landing page (TSX)
│   └── css/          — theme overrides
├── static/
│   ├── img/          — logo / favicon
│   └── schema/       — generated query-plan JSON Schema
├── docusaurus.config.ts
├── sidebars.ts
├── package.json
└── tsconfig.json
```

The pages are routed via the explicit sidebar list in
`sidebars.ts` (not autogenerated from filenames), so adding a new
page **also** requires editing the sidebar.

## Local preview

```bash
cd docs-site
npm ci          # first run only / after package.json changes
npm run start   # http://localhost:3000, hot-reloads on save
```

Use `npm run build` (followed by `npm run serve`) to test the
production build locally.

## Regenerate reference fragments

`scripts/generate_docs_reference.py` is the source of truth for:

- the `QueryPlan` JSON Schema at
  `docs-site/static/schema/query-plan.schema.json`;
- the managed config table inside
  `docs-site/docs/reference/configuration-reference.md`;
- the managed tool / resource / prompt tables inside their
  respective `*-reference.md` pages.

When to run it:

| Change | Run? |
| --- | --- |
| Edited `.env.example` | yes |
| Changed any IR class in `src/graph_mcp/models/` | yes |
| Added / renamed / removed an `@mcp.tool()` | yes |
| Added / renamed / removed an `@mcp.resource(...)` | yes |
| Added / renamed / removed an `@mcp.prompt(...)` | yes |
| Pure prose change to a docs page | no |

```bash
python scripts/generate_docs_reference.py          # regenerate
python scripts/generate_docs_reference.py --check  # CI-style stale check
```

CI runs the `--check` form. Stale fragments fail the docs workflow
with a unified diff showing exactly what changed.

## Update the QueryPlan JSON Schema

The schema is exported from
`pydantic.TypeAdapter(QueryPlan).json_schema()` whenever you run the
generator. There is no separate schema file to hand-edit.

If you change an IR class:

1. Update the model.
2. Run `python scripts/generate_docs_reference.py`.
3. Diff `docs-site/static/schema/query-plan.schema.json` against
   what was committed; commit only the deliberate changes.

## Add a new user-guide page

1. Create `docs-site/docs/users/<slug>.md` with frontmatter:
   ```yaml
   ---
   id: <slug>
   title: <Title>
   sidebar_position: <number>
   description: <one-sentence purpose>
   ---
   ```
2. Add `users/<slug>` to the `users` sidebar in
   `docs-site/sidebars.ts`.
3. Cross-link from at least one neighbouring page so the doc is
   reachable without using the sidebar.
4. Avoid placeholder markers in committed prose — the doc-coverage
   checker (`scripts/check_docs.py`) rejects them outside the
   explicitly allowed sections (`Known limitations`, `Future work`,
   `Roadmap`, `Open questions`). Run the checker locally to see the
   exact list it scans for.

## Add a new reference page

1. Create `docs-site/docs/reference/<slug>.md` (same frontmatter
   shape).
2. Append `reference/<slug>` to the `reference` sidebar in
   `docs-site/sidebars.ts`.
3. If the page should contain an auto-derived list of items, add a
   managed block:
   ```md
   <!-- BEGIN: managed:<table-name> -->
   ...
   <!-- END: managed:<table-name> -->
   ```
   Then teach `scripts/generate_docs_reference.py` how to render the
   table and add the corresponding plan in `build_plans()`.
4. If the new page enumerates registered MCP surfaces, also extend
   `scripts/check_docs.py` so missing entries fail the build.

## Add a new ADR

ADRs are short, numbered decision records under
`docs-site/docs/adr/`.

1. Pick the next sequential number.
2. Copy `0004-docusaurus-documentation-site.md` as a template.
3. Use the canonical sections: **Context**, **Decision**,
   **Alternatives considered**, **Consequences**, optionally
   **References**.
4. Add the file to the `adr` sidebar in `docs-site/sidebars.ts`.
5. Cross-link from any developer page where the decision is
   load-bearing.

## Update workflows

- `.github/workflows/docs.yml` — builds + deploys the site to
  GitHub Pages on push to `main`. Path filters here need to mention
  every input the docs depend on (config, IR sources, scripts).
- `.github/workflows/ci.yml` — independently runs
  `generate_docs_reference.py --check`, `check_docs.py`, and a
  Docusaurus build, so PRs fail fast even when the dedicated docs
  workflow's path filters do not match.

When you add a new generated artifact, update both workflows.

## Debug broken Docusaurus links

`docusaurus.config.ts` sets `onBrokenLinks: "throw"` and
`markdown.hooks.onBrokenMarkdownLinks: "throw"`, so any broken link
fails the build. The error message names the offending file and the
unresolved target.

Common fixes:

- relative links to `../foo.md` should become absolute Docusaurus
  routes (`/users/foo/`) — the site uses `routeBasePath: "/"` so
  `/users/foo/` is the correct shape.
- anchor links must match an existing heading slug (lowercased,
  hyphenated). When in doubt, look at the rendered headings in
  `npm run build`'s output or visit the page in `npm run start`.
- managed-table anchors are checked by
  `scripts/check_docs.py:check_anchor_consistency`.

## Publish via GitHub Pages

The `docs.yml` workflow deploys on every push to `main` via
`actions/deploy-pages@v4`. The repository must have **Settings →
Pages → Build and deployment → Source** set to **GitHub Actions**;
without it, the deploy step cannot publish even if the workflow runs
successfully.

`docusaurus.config.ts` derives `url` and `baseUrl` from
`GITHUB_REPOSITORY`; for custom domains, set
`DOCUSAURUS_URL` and `DOCUSAURUS_BASE_URL` as repository Variables
and the workflow forwards them.

## What CI checks must pass

For a docs PR to merge, both jobs (in `ci.yml` and `docs.yml`) must
pass:

- `python scripts/generate_docs_reference.py --check` — no stale
  managed fragments;
- `python scripts/check_docs.py` — coverage, anchors, forbidden
  URLs, no unauthorized placeholders;
- `npm run typecheck` — TypeScript config and TSX page compile;
- `npm run build` — Docusaurus build succeeds with
  `onBrokenLinks: "throw"`.

If a hand-written page goes out of date but isn't auto-checked
(e.g. an architecture diagram referencing renamed modules), the
checks won't catch it. Treat the developer/architecture pages as
hand-maintained and review them whenever module structure changes.
