# ADR 0004 — Docusaurus documentation site

**Status:** accepted

## Context

`graph-mcp` started as a single-README repo. The README has grown to
~440 lines covering installation, configuration, tools, resources,
security, the eval harness, and limitations. It is too dense for new
users and too shallow for contributors who need internals.

We need a documentation site that:

- separates user-facing guidance from internals;
- supports a stable reference layer (config, tools, resources,
  validation errors, eval metrics) that can be auto-derived;
- carries Architecture Decision Records;
- ships in the same repo so docs PRs ride alongside code PRs;
- deploys for free on GitHub Pages;
- builds in CI so documentation drift is caught at PR time.

## Decision

Use [Docusaurus 3](https://docusaurus.io) as the static site
generator, with TypeScript configuration, Mermaid for diagrams, and
[`@easyops-cn/docusaurus-search-local`](https://github.com/easyops-cn/docusaurus-search-local)
for offline search. Deploy to GitHub Pages via
`actions/deploy-pages@v4` from a dedicated `.github/workflows/docs.yml`
workflow.

## Alternatives considered

### A. MkDocs Material

Pros: small Python-native dep tree (matches the rest of the project),
fast builds, excellent Material theme, easy local search. Cons:
JSDoc/MDX integration is awkward; less out-of-the-box support for
versioned docs; team is more familiar with Docusaurus.

Considered seriously. The deciding factor was Docusaurus's first-class
support for versioning if/when we need it, and the cleaner MDX story
for the JSON Schema and API examples we want to render inline.

### B. VitePress

Pros: very fast, Vue-based, minimal. Cons: smaller ecosystem of
plugins; less convention around docs/blog/sidebar separation; team
familiarity is lower.

Considered, but the missing pieces (search, Mermaid, sidebar shape)
would need integration work.

### C. Sphinx

Pros: the de-facto standard for Python projects; first-class
docstring extraction. Cons: reST primary syntax (Markdown via
extensions); themes feel dated; Mermaid integration is third-party;
and our docs are mostly hand-written narrative, not API reference.

Considered but rejected — the docs we are writing are not Python API
docs. They are user guides, developer guides, and reference tables.

### D. Plain Markdown + GitHub README

Pros: zero build. Cons: no sidebar, no search, no diagrams, no
cross-doc linking; the README is already too long.

Rejected as the status quo we're moving away from.

### E. Docusaurus (chosen)

Pros: well-supported, MDX, sidebar, versioning, search via plugin,
official GitHub Pages support, large community. Cons: Node toolchain
(separate from the Python project), TypeScript boilerplate, build is
slower than VitePress.

The Node toolchain is the largest cost. We mitigate by isolating it
to `docs-site/` with its own `package.json`, lockfile, and CI job.

## Why GitHub Pages via Actions

- Free for public repositories;
- official actions (`actions/configure-pages`,
  `actions/upload-pages-artifact`, `actions/deploy-pages`) handle the
  artifact lifecycle;
- OIDC-based deploy means no long-lived secrets;
- the `pages` concurrency group and `cancel-in-progress: false`
  prevent overlapping deployments from fighting.

## Consequences

- Adds a Node toolchain to the repository. Mitigated by isolating it
  to `docs-site/`.
- Docs become part of the build matrix: PRs that touch docs build the
  site; PRs that touch source code that the docs reference (config,
  tools, resources, IR schema) trigger the docs workflow.
- Reference fragments are generated by
  `scripts/generate_docs_reference.py` so the configuration and
  tool/resource lists never drift from the runtime. The `--check`
  mode is a CI gate.
- A second CI script, `scripts/check_docs.py`, fails the build when a
  newly added env variable, MCP tool, or MCP resource is missing
  from the reference pages.
- The site supports Mermaid diagrams without extra wiring.
- Versioning is available if/when we cut the first stable release;
  not enabled now.
